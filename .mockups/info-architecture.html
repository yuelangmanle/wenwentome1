第一部分设计 / 信息架构

书库中台型阅读器

你的 app 不把“本地书”和“网文”做成两套系统，而是让它们先汇入统一书目模型， 再进入同一个书库、同一个详情页、同一个阅读器。

书库是首页，不是发现页 因为你第一阶段最重视的是书库管理，所以用户打开 app 应该先看到自己的书，而不是推荐流。
发现页只负责“入库” 网文搜索、内置源、规则导入都放在发现里，但找到书之后动作是加入书库，而不是长期停留在发现页消费。
统一对象，来源只做徽标 用户看到的是一本书，不是一个来源类型。来源差异只在卡片徽标和详情能力里轻量表达。

为什么不是“双系统拼合”

你最重视的是书库管理

如果本地书和网文底层是两套模型，筛选、排序、同步、书签恢复会越来越割裂。

你要的是二合一整合感

统一书库能让“导入 EPUB”和“加入网文”都变成同一种用户动作：把一本书放进自己的书架。

旧项目只参考，不决定主架构

搜书大师和旧阅读系可以借鉴交互和规则引擎，但不该把你的主工程拖回旧式安卓大应用结构。

第二部分设计 / 模块边界

哪些自己做，哪些借力

按我们前面定的 `混合路线`，主工程和产品层必须是你自己的，网文规则引擎则做成单独桥接层。 这样既能保住 `书库管理优先` 和 `Compose 架构`，也不会把“阅读系书源兼容”从零重写一遍。

自己做 App Shell

应用壳层

负责导航、权限申请、导入入口、页面状态恢复、全局主题和错误提示，是整个产品体验的门面层。

负责：底部导航、详情页跳转、导入动作、全局设置入口
原因：这是你产品感最强的一层，不能被旧项目结构绑架
自己做 Library Core

书库中台

这是第一阶段的核心。所有书都要先落进这里，再谈阅读、同步和来源差异。

负责：统一书目、分组筛选、搜索、最近阅读、批量操作
原因：你最重视的是书库管理，这层必须按你的产品目标来设计
自己做 Reader UX

阅读体验层

统一详情页、统一阅读页、统一书签和进度体验。第一阶段先围绕 TXT / EPUB 做可重排阅读。

负责：阅读页、目录、书签、主题、翻页、进度恢复
原因：用户感知最强，后期也最需要持续迭代
自己做 Sync Engine

GitHub 同步层

用私有仓库同步书文件和元数据，这属于你产品的独特约束，旧阅读项目不会天然帮你做好。

负责：同步清单、文件上传下载、冲突策略、恢复与重建
原因：这是你明确提出的核心能力，而且会影响数据模型
桥接层 Source Bridge

网文规则桥

单独做成 `source-engine` 或 `source-bridge` 模块，向上暴露统一接口，向下适配阅读系规则能力。

负责：搜索、详情、目录、正文、Cookie、规则执行、源导入
原因：这层变化最大，隔离开后才不会污染主工程
借鉴/复用思路 Legacy Engine

旧阅读系能力参考

这里优先参考 `Legado / BookshelfLib / 搜书大师` 的规则边界和旧模型，不直接把整个 app 搬进来。

借：规则能力、字段语义、旧源兼容经验
不借：旧 UI 架构、商业化壳层、社区和广告模块
主工程边界

主工程只认识“书”和“阅读状态”，不直接依赖某一种书源格式。

桥接层边界

桥接层负责把规则世界翻译成统一书目对象，向上屏蔽旧源差异。

同步边界

同步层只同步标准化后的数据和文件，不关心书来自本地还是网文。

第二部分设计 / 数据模型骨架

先统一“书”，再统一“来源”

第一阶段的数据模型不应该先从书源规则出发，而应该先定义统一书目。 这样本地 TXT / EPUB 和网文都能进同一个书库；来源能力是挂在书上的扩展信息，不是顶层对象本身。

核心实体

BookRecord

书库里的一本书，不区分本地书还是网文，是用户真正能看到和管理的对象。

id：统一书 ID
title / author：标准书目字段
originType：local / web / mixed
primaryFormat：txt / epub / web
bookshelfState：在架、归档、收藏、最近阅读
核心实体

BookAsset

书对应的实体文件或缓存对象，方便同步、恢复和多来源挂接。

bookId：归属哪本书
storageUri：本地文件位置
mime / size / hash：文件校验信息
syncPath：GitHub 仓库中的存放路径
assetRole：正文文件 / 封面 / 缓存包
核心实体

ReadingState

不管来源如何，阅读进度、书签、最近章节、主题偏好都在这里统一表达。

bookId：归属哪本书
locator：本地页码 / EPUB CFI / 章节锚点
chapterRef：当前章节引用
bookmarks：书签和批注
updatedAt：冲突比较基准
桥接实体

SourceDefinition

书源本身。属于桥接层管理对象，不应该直接污染书库主模型。

sourceId：源唯一标识
sourceType：内置 / 导入 / 兼容源
ruleFormat：legacy / legado3 / custom
authState：Cookie / 登录状态
enabled：是否参与搜索与抓取
桥接实体

RemoteBinding

一本书和某个网文源的绑定关系。用来表示它在来源世界里的身份，不等于书本身。

bookId：归属哪本书
sourceId：来自哪个书源
remoteBookId：源站内 ID / URL
tocRef：目录入口
syncMode：追更 / 手动刷新 / 只读缓存
同步实体

SyncSnapshot

GitHub 私有仓库中的标准化同步单元，服务于恢复、冲突处理和跨设备一致性。

snapshotId：一次同步快照
manifest：书目与资产索引
revision：仓库提交或版本号
deviceId：来源设备
mergedAt：最后一次合并时间
这套骨架的关键含义

`BookRecord` 是第一公民；`SourceDefinition` 和 `RemoteBinding` 只是让网文能力接进来。 这样本地书不会沦为“兼容项”，网文也不会被排除在统一书库之外。

第三部分设计 / 用户流程与验证

先把主链路做稳，再让复杂能力渐进出现

第一阶段虽然是二合一，但真正要打穿的是三条清晰旅程：`本地导入入库`、`网文加入书库`、`GitHub 同步恢复`。 只要这三条稳住，后续再加 PDF、复杂书源、重登录流程都不会把主架构推倒。

主链路 A

本地导入入库

这是第一阶段最核心的体验，必须做到“导入后立刻能管理、能读、能同步”。

1. 用户从书库页点击导入，选择文件或扫描目录
2. 解析 TXT / EPUB 元数据，生成 `BookRecord + BookAsset`
3. 自动进入统一书库，可立即分组、搜索、开始阅读
4. 后台按同步策略把文件和元数据推到 GitHub 私有仓库
主链路 B

网文加入书库

发现页不是消费终点，而是“把远端书目绑定进我的书库”的入口。

1. 用户在发现页搜索或浏览来源
2. 规则桥获取详情、目录和可读性信息
3. 点击加入书库，生成 `BookRecord + RemoteBinding`
4. 用户回到书库后，看到它和本地书处在同一套管理体系里
主链路 C

GitHub 恢复与换机

这是你要“少装东西、轻运维”的关键补偿，必须让换设备后继续读变成自然动作。

1. 新设备登录或配置 GitHub 私有仓库
2. 拉取 `SyncSnapshot` 清单并重建书库索引
3. 按需下载书文件和封面，恢复书签、进度、设置
4. 用户打开任一本书，直接从上次位置继续
错误处理

异常设计原则

错误不能直接暴露为“失败了”，而要明确告诉用户是导入问题、源问题、同步问题，还是权限问题。

导入失败：告诉用户是格式不支持、文件损坏，还是编码识别失败
书源失败：区分“源失效 / 登录失效 / 规则不兼容 / 网络异常”
同步失败：区分“认证失败 / 冲突 / 仓库不可达 / 文件过大”
渐进策略

复杂能力延后出现

第一阶段不要一上来就追求把所有复杂源和所有同步边界都吃满，而是先保证主链路稳定。

先做：TXT / EPUB、本地导入、书库管理、可用同步、常见规则子能力
后补：PDF、复杂聚合源、深登录态、发现页花样玩法、重缓存策略

解析测试

验证 TXT / EPUB 解析、封面提取、章节切分、元数据标准化。

规则契约测试

用代表性书源样本验证搜索、详情、目录、正文四类规则是否还能跑通。

同步回归测试

验证文件上传下载、断点恢复、冲突合并、换机恢复和弱网重试。

UI 冒烟测试

验证书库、详情、阅读页、发现页、设置页的核心跳转和状态保持。

第三部分的核心判断

如果你认可这三条主链路和这个测试边界，设计阶段其实就基本闭合了。 下一步就不是继续发散，而是进入 `write design doc`，把已经确认的内容固化成正式规格文档。
