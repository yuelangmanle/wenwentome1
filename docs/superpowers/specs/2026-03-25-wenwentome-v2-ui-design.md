# 文文tome V2 UI 与交互重设计方案

## 1. 背景

当前 `v2.0.0` 已完成底座迁移与云端正式发版，但运行时 UI 仍然主要沿用 Legado 现有结构。用户已经明确反馈以下问题：

- 阅读页正文区域被工具面板挤占，不够沉浸
- `发现` 页的网文能力、书源管理与后续浏览器找书能力边界不清晰
- 导入、下载缓存管理、书籍详情、换源等高频动作的入口不够统一
- `AI / 同步` 需要保留显眼入口，但不应该干扰书架和阅读主链

本方案的目标不是复用旧版 `Compose` UI，而是基于当前功能现状重新设计一套新的信息架构与交互层级。

## 2. 设计目标

### 2.1 产品定位

- 产品气质：`均衡型`
- 不是纯阅读器，也不是工具箱
- 阅读主链必须清晰
- 发现、导入、同步、AI 也必须顺手可达

### 2.2 核心原则

- 底部导航只保留真正的一层主模块
- 一个页面只解决一类核心任务，避免“首页混一切”
- 阅读默认态必须正文优先
- 不管书从哪里来，阅读、详情、下载、缓存管理尽量统一
- 高频动作靠近主操作区，低频管理收进 `...`

## 3. 顶层信息架构

底部导航固定为 3 个一级入口：

1. `书架`
2. `发现`
3. `我的`

不再单独给 `导入` 一个底部入口，也不把 `AI / 同步` 放在首页混排。

### 3.1 书架

`书架` 就是纯书架页，不承担发现、AI、同步等跨域功能。

主内容：

- 继续阅读
- 书籍列表 / 书架分组
- 排序 / 筛选

右上角 `...` 菜单：

- 导入书籍
- 下载缓存管理
- 批量管理
- 排序 / 筛选

### 3.2 发现

`发现` 专门承接网文相关功能。

顶部采用双入口切换：

- `书源搜索`
- `浏览器找书`

右上角只有一套 `...`，且为联动菜单：

- 在 `书源搜索` 模式显示书源相关项
- 在 `浏览器找书` 模式显示浏览器与搜索引擎相关项

### 3.3 我的

`我的` 页顶部保留两个显眼能力入口：

- `AI`
- `同步`

其下再放：

- 设置
- 关于
- 版本信息
- 高级功能

这样可以保证 `AI / 同步` 可见，但不干扰书架主链。

## 4. 书架页设计

### 4.1 页面角色

书架页只承担“管理现有书”和“继续阅读”。

不在书架首页展示：

- AI 总览
- 同步状态大卡
- 浏览器找书
- 书源管理

### 4.2 导入入口

导入功能放到 `书架 > 右上角 ... > 导入书籍`。

导入书籍再分两类：

- 扫描本地书籍
- 按路径手动导入

### 4.3 扫描本地书籍

必须采用流式扫描，而不是等待全盘结束后一次性返回。

用户体验要求：

- 扫到一本显示一本
- 用户可边看边勾选或直接导入
- 不需要等全部扫描结束
- 支持中途暂停 / 停止 / 继续

### 4.4 下载缓存管理入口

统一入口为：

`书架 > 右上角 ... > 下载缓存管理`

不允许不同来源分别拥有不同缓存页。

## 5. 发现页设计

## 5.1 书源搜索

定义：

- `书源搜索` = 使用已导入并启用的书源进行网文搜索

页面职责：

- 输入关键词
- 基于当前启用书源搜索
- 展示结果
- 进入书籍详情页
- 再进入阅读或加入书架

右上角 `...` 菜单在该模式下显示：

- 书源管理
- 导入书源
- 启用 / 禁用书源
- 删除书源

### 5.2 书源搜索结果详情

书源搜索得到的书，点击后先进入 `书籍详情页`，而不是直接进入阅读。

详情页至少展示：

- 书名
- 作者
- 简介
- 封面
- 来源信息
- 加入书架
- 开始阅读

这是书源网文链路的标准入口。

### 5.3 浏览器找书

定义：

- `浏览器找书` = 使用内置浏览器能力与搜索引擎在全网找书，再识别网页并转化为可阅读内容

#### 页面结构

顶部是一条浏览器搜索栏。

搜索栏左侧：

- 当前搜索引擎标签
- 例如：`必应`

点击后弹出快速切换面板，用于高频切换搜索引擎。

首批内置搜索引擎：

- 必应
- 百度
- 搜狗
- 搜狐
- 神马
- 谷歌
- 自定义

#### 右上角 `...`

在 `浏览器找书` 模式下，右上角统一菜单显示：

- 搜索引擎管理
- 切换搜索引擎
- 自定义搜索引擎
- 切换浏览器模式
- 浏览器找书设置

注意：

- 搜索引擎“快速切换”由搜索栏左侧标签负责
- 右上角 `...` 负责低频管理和设置

### 5.4 浏览器模式

浏览器找书提供 3 种浏览器模式：

1. `标准浏览`
2. `沉浸找书`
3. `阅读优化优先`

默认模式为 `标准浏览`。

定义如下：

- `标准浏览`
  - 保持正常网页浏览
  - 适合搜索、点目录、看站点结构
- `沉浸找书`
  - 弱化站点杂项 UI
  - 强调目录、正文、下一章、加入书架等识别结果
- `阅读优化优先`
  - 只要识别到正文就优先转为阅读器样式

### 5.5 优化阅读悬浮窗

浏览器找书模式下保留一个悬浮窗，用于阅读优化控制。

浮窗点击后可配置：

- 自动优化阅读
- 关闭自动优化，改为手动

手动模式下：

- 长按悬浮窗触发优化阅读

浮窗状态建议为：

- 灰色：当前页未识别到可优化内容
- 高亮：当前页已识别，可一键优化
- 常亮-A：自动优化已开启

## 6. 统一阅读页设计

### 6.1 默认态

阅读页采用 `纯阅读默认态`。

即：

- 默认只给正文
- 工具不常驻
- 点击页面后再呼出目录、设置、进度、AI、换源、下载

目标：

- 正文区域最大化
- 工具不挤占阅读空间

### 6.2 阅读页统一右上角菜单

所有来源的书，在阅读页右上角都保留菜单能力。

来源包括：

- 本地书籍
- 书源搜索加入书架的书
- 浏览器找书识别并入库的书

阅读页右上角菜单统一包含：

- 书籍详情
- 下载
- 换源

## 7. 书籍详情策略

### 7.1 来源差异

- `书源搜索` 找到的书：
  - 先天拥有详情页
- `浏览器找书` 找到的书：
  - 不强制先走完整详情页
  - 可直接进入阅读或轻量识别结果页
- `本地书`
  - 进入阅读后仍然允许从右上角菜单进入书籍详情

### 7.2 统一书籍详情

一旦书已经进入书架或阅读体系，详情体验应统一：

- 都能从阅读页右上角进入 `书籍详情`
- 如果本地信息不足，则自动联网搜索和补齐信息

联网补全目标：

- 作者
- 简介
- 封面
- 标签 / 分类

即：

- 书源书优先使用书源已有信息
- 本地书、浏览器书若信息不足，则执行联网补全

## 8. 统一下载与缓存管理

### 8.1 阅读页下载按钮

以下来源的书，进入阅读页后右上角都保留 `下载` 按钮：

- 书源导入后的网文书
- `书源搜索` 找到并加入的书
- `浏览器找书` 识别并转成可阅读的书

### 8.2 下载范围

下载策略统一支持：

- 下载后面 `N` 章
- 下载后面全部
- 下载整本书

### 8.3 浏览器找书来源的抓取规则

浏览器找书来源的书：

- 优先识别目录
- 若无目录，则继续识别：
  - 下一章
  - 下一页
  - 连续翻页链

用户已确认：

- 默认按当前站点可识别的翻页链继续抓取

### 8.4 缓存管理页

下载缓存管理页统一展示：

- 封面
- 书名
- 来源类型
- 缓存状态
- 已缓存章节数 / 总章节数
- 最后更新时间
- 继续下载
- 暂停
- 删除缓存

### 8.5 封面策略

封面必须保留：

- 书架保留
- 下载缓存管理页保留
- 浏览器找书来源尽量抓取网页主图 / 封面作为封面

## 9. 换源设计

### 9.1 适用范围

`书源搜索` 找到并入库的书必须支持换源。

### 9.2 两层换源

换源拆成两层：

1. `阅读页快速换源`
2. `书籍详情页整书换源`

#### 阅读页快速换源

用于处理：

- 当前章节可读
- 下一个章节失效
- 某章临时失效

支持：

- 当前章节换源
- 后续章节换源

#### 书籍详情页整书换源

用于处理：

- 长期切换到更稳定的来源
- 整个目录、封面、最新章来源切换

### 9.3 自动 / 手动换源

阅读链路允许后续扩展为：

- 手动换源
- 自动候选源兜底

但本方案优先保证：

- 用户明确可手动换源
- 章节级失效时可继续阅读

## 10. 技术路线建议

### 10.1 总路线

不直接集成一整个完整浏览器 App。

推荐路线：

- `System WebView` 作为浏览器找书容器
- `Mozilla Readability` 用于正文提取与优化阅读
- 自己实现搜索引擎模板管理
- 自己实现小说站目录 / 正文 / 翻页 / 下一章识别层

### 10.2 搜索引擎管理

搜索引擎数据模型建议支持：

- 名称
- 搜索 URL 模板
- `%s` 关键词占位符
- 可选编码
- 可选 User-Agent

### 10.3 不推荐路线

不建议直接把以下项目整包嵌入：

- Lightning Browser
- Fulguris
- Firefox Android 整包

原因：

- 它们是完整浏览器产品
- 你当前需要的是“发现页里的浏览器找书能力”
- 整包集成会引入过多与阅读器无关的浏览器负担

## 11. 实施优先级建议

### Phase 1：信息架构与入口重排

- 重做底部导航
- 重做书架 `...`
- 重做发现页双入口
- 重做我的页 AI / 同步入口

### Phase 2：浏览器找书 MVP

- WebView 容器
- 搜索引擎切换
- 自定义搜索引擎
- 优化阅读悬浮窗
- Readability 接入

### Phase 3：统一阅读与缓存

- 阅读页统一右上角菜单
- 下载按钮统一化
- 下载缓存管理页统一
- 浏览器找书翻页抓取链

### Phase 4：详情与换源闭环

- 统一书籍详情
- 联网补全元数据
- 阅读页快速换源
- 整书换源

## 12. 结论

本方案最终把产品收敛为：

- `书架` 负责现有书与继续阅读
- `发现` 负责网文发现，两条链路分别是 `书源搜索` 与 `浏览器找书`
- `我的` 负责 AI、同步、设置和关于
- 阅读页保持纯阅读默认态
- 所有来源的书统一进入详情、阅读、下载、缓存管理体系

这是一套“均衡型”的新 UI 方案，不复用旧版 UI，不继续放大当前 Legado 原始界面的历史结构，而是在当前功能基础上重新梳理出适合文文tome 的产品层交互。
