# 书源兼容底座第一阶段设计

## 1. 背景

WenwenToMe 当前已经具备本地书阅读、网文发现、目录拉取、正文展示、阅读进度保存和 GitHub 云端发版链路，但网文书源兼容底座仍然明显偏弱。

现状问题集中在以下几类：

1. 规则支持范围过窄
   - 当前桥接层主要支持普通 `HTML + CSS selector`
   - 遇到 `$` 开头的 JSONPath 规则、`##` 文本清洗、`@js:` 后处理时要么直接跳过，要么静默退化
   - `<js>` 搜索模板、登录态、聚合脚本源几乎不可用
2. 兼容逻辑分散
   - `SourceBridgeRepository` 直接承载搜索、详情、目录、正文四条业务链
   - `JsoupRuleExecutor` 当前通过字符串截断规避复杂语法，不适合作为长期底座
3. 缺少真实验收闭环
   - 现有测试更多是语法或局部流程测试
   - 缺少“某个真实源从搜索到正文是否完整可读”的固定验收清单

因此，`1.4` 第一阶段不追求一次性兼容完整阅读类书源生态，而是优先建立一个可扩展的书源兼容底座，并确保第一批真实源可以稳定跑通 `搜索 -> 详情 -> 目录 -> 正文` 主链。

## 2. 本轮目标

### 2.1 产品目标

- 让一批真实网文书源从搜索到阅读真正可用
- 为后续继续兼容复杂书源语法打下结构化底座
- 明确记录“第一批支持什么，不支持什么”，避免范围失控

### 2.2 已确认的用户决策

- 本轮优先级选择：`完整阅读闭环优先`
- 第一批复杂度边界：`中等复杂度`
- 第一批支持范围：
  - HTML / 普通规则
  - `##` 文本清洗
  - 轻量 `@js:` 字段后处理
  - `$` 开头的 JSON 列表规则
- 第一批验收目标：`10 个可用源`
- 第一批暂不纳入：
  - `<js>` 搜索模板
  - 登录态 / cookie / 用户书架类源
  - 强依赖远端聚合脚本服务的源

## 3. 设计原则

### 3.1 闭环优先，不做语法堆砌

本轮成功标准不是“又多支持了几个语法关键字”，而是“有一批真实书源可以完整阅读”。

### 3.2 扩展现有底座，而不是推翻重写

项目已经具备 `JsoupRuleExecutor`、`XPathRuleExecutor`、`JsonPathRuleExecutor`、`RhinoScriptEvaluator` 等基础能力，本轮采用增量扩展，而不是重做统一 DSL/AST 解释器。

### 3.3 语法解析与业务流程解耦

规则字符串的拆解、规则执行、业务模型组装分层处理，避免把搜索、详情、目录、正文四条流程都塞进同一个仓库类。

### 3.4 明确边界

对第一批不支持的规则类型要显式识别并给出诊断，不以“空结果”掩盖能力边界。

## 4. 方案对比

### 方案 A：现有桥接层增量扩展（推荐）

保留当前 `SourceBridgeRepository + SourceRuleParser + LegacySourceMapper + 执行器` 结构，在中间加入结构化规则对象、规则拆解器和统一执行器。

优点：

- 改动集中，适合当前代码库
- 能最快做出首批 10 个可用源
- 后续继续扩 `@js:`、`<js>` 时仍有演进空间

缺点：

- 结构会比当前实现复杂一些
- 后续支持更重的脚本语法时仍需再扩一层

### 方案 B：重做统一 DSL / AST 解释器

优点：

- 长期结构最整齐
- 理论上最适合完整兼容复杂书源生态

缺点：

- 前期投入过大
- 不适合当前“先收口真实可用源”的目标

### 方案 C：按站点写特化适配器

优点：

- 短期见效快

缺点：

- 不可扩展
- 与“兼容书源生态”方向相反
- 会让后续维护成本迅速恶化

结论：采用 `方案 A`。

## 5. 范围定义

### 5.1 第一批纳入范围

- 搜索页：
  - HTML 列表搜索
  - JSON 列表搜索
- 详情页：
  - HTML 详情抽取
  - JSON 字段抽取（若目标源需要）
- 目录页：
  - HTML 目录抽取
  - JSON 目录抽取（若目标源需要）
- 正文页：
  - HTML 正文抽取
  - `##` 文本清洗
  - 轻量 `@js:` 文本后处理

### 5.2 明确不做

- `<js>` 请求模板
- 登录态 / cookie / 用户个人书架类源
- Explore 发现页复杂脚本源
- 多页目录自动翻页
- 多页正文自动拼接
- 重度加密、动态签名、需要 app 运行时环境对象的源

## 6. 总体架构

第一批把书源兼容底座拆成四层：

1. `源解析层`
   - 从旧书源 JSON 解析出标准源定义
   - 保留复杂规则信息，不在这里做静默裁剪
2. `规则执行层`
   - 根据规则引擎选择 `CSS / XPath / JSONPath`
   - 统一处理列表抽取、字段抽取、清洗和轻量后处理
3. `后处理层`
   - 这里不是独立公开组件，而是 `RuleExecutionEngine` 内部的固定阶段
   - 负责 `##` 清洗、轻量 `@js:` 和字段值归一化
4. `验收层`
   - 固定维护首批目标源名单与测试样本

## 7. 数据模型设计

### 7.1 现状问题

当前 `NormalizedSourceDefinition` 中的规则字段大多是裸字符串，例如：

- `searchUrlTemplate`
- `SearchRule.bookList`
- `SearchRule.name`
- `ContentRule.content`

这种结构无法表达：

- 当前规则是 CSS / XPath / JSONPath 的哪一种
- 是否带 `##` 清洗
- 是否带 `@js:` 后处理
- 字段提取后是否需要再归一化

### 7.2 新模型方向

在不推翻现有模型的前提下，为字段规则引入结构化表达。

建议新增：

- `RuleEngine`
  - `HTML_CSS`
  - `HTML_XPATH`
  - `JSON_PATH`
- `ResponseKind`
  - `HTML`
  - `JSON`
- `RuleExpression`
  - `engine`
  - `selector`
  - `extractor`
  - `cleaners`
  - `postScript`
- `TextCleaner`
  - `RemoveRegex`
  - `ReplaceRegex`

业务规则对象例如：

- `SearchRule`
- `BookInfoRule`
- `TocRule`
- `ContentRule`

内部不再只保留原始字符串，而是改为使用 `RuleExpression`。

## 8. 规则拆解设计

### 8.1 目标

把旧书源里的规则字符串拆成可执行结构，而不是在执行阶段再靠字符串截断临时判断。

### 8.2 拆解职责

新增一个独立的 `RuleExpressionParser`，负责：

- 判断引擎类型
- 提取主表达式
- 提取 `##` 清洗链
- 提取 `@js:` 后处理脚本

### 8.3 引擎判断规则

- `$` 开头：`JSON_PATH`
- `//` 开头：`HTML_XPATH`
- 其他：默认 `HTML_CSS`

### 8.4 `##` 规则

第一批只支持两类高频语义：

1. `rule##regex`
   - 表示用正则删除命中的内容
2. `rule##regex##replacement`
   - 表示做一次正则替换

这两类足够覆盖正文清洗、章节名去广告、标题去站名等第一批高频场景。

### 8.5 `@js:` 规则

第一批只支持字段级后处理：

- 脚本形态限定为“单个表达式”，不支持多语句流程控制脚本
- 可访问绑定只包含：
  - `result`
  - `baseUrl`
  - `pageUrl`
- 执行器使用受限的 `Rhino` 安全作用域，不注入 `java`、`source`、`cookie`、网络能力或其他 app 运行时对象
- 输出规则：
  - 表达式结果统一按字符串处理
  - `null`、`undefined` 或空白结果回退为原始 `result`
- 异常规则：
  - 任意脚本异常统一记为 `SCRIPT_POST_PROCESS_FAILED`
  - 当前字段回退为脚本执行前的原始值
  - 不重试，不中断同源其他字段解析

第一批不允许：

- 发网络请求
- 访问 `java/source/cookie` 等 app 运行时对象
- 修改流程控制
- 使用需要循环、外部状态或多步副作用的脚本

### 8.6 `<js>` 规则

解析阶段识别，但执行阶段直接标记为“超出第一批范围”。

## 9. 统一执行器设计

### 9.1 目标

避免 `SourceBridgeRepository` 直接操作三套执行器和字符串处理逻辑。

### 9.2 新增组件

新增 `RuleExecutionEngine`，职责如下：

- 接收 `RuleExpression`
- 根据 `engine` 选择执行器
- 处理列表抽取与字段抽取
- 在引擎内部按固定顺序执行：
  - 原始取值
  - `##` 清洗链
  - `@js:` 轻量后处理
  - 结果归一化
- 返回最终字符串结果或列表结果

`SourceBridgeRepository` 不直接执行清洗或脚本，只负责请求调度、页面上下文准备和业务模型组装。

### 9.3 底层执行器复用

- `JsoupRuleExecutor`
  - 负责 CSS selector
- `XPathRuleExecutor`
  - 负责 XPath
- `JsonPathRuleExecutor`
  - 负责 JSONPath
- `RhinoScriptEvaluator`
  - 负责轻量字段级脚本

## 10. 运行时执行流

### 10.1 搜索

1. 读取源定义
2. 解析搜索 URL 模板
3. 发请求
4. 根据规则判断响应按 `HTML` 还是 `JSON` 处理
5. 执行 `bookList`
6. 对每个 item 依次执行：
   - `name`
   - `bookUrl`
   - `author`
   - `coverUrl`
   - `intro`
7. 每个字段统一交给 `RuleExecutionEngine` 完成取值、清洗、后处理和结果归一化
8. 组装 `RemoteSearchResult`

### 10.2 详情

1. 请求详情页
2. 按源定义选择 `HTML` 或 `JSON`
3. 依次抽取：
   - `name`
   - `author`
   - `intro`
   - `coverUrl`
   - `kind`
   - `lastChapter`
4. 每个字段统一交给 `RuleExecutionEngine` 完成取值、清洗、后处理和结果归一化
5. 组装 `RemoteBookDetail`

### 10.3 目录

1. 请求目录页
2. 执行 `chapterList`
3. 对每个 item 抽取：
   - `chapterName`
   - `chapterUrl`
4. 每个字段统一交给 `RuleExecutionEngine` 完成取值、清洗、后处理和结果归一化
5. 组装 `RemoteChapter`

### 10.4 正文

1. 请求正文页
2. 执行 `content`
3. 正文字段统一交给 `RuleExecutionEngine` 完成取值、`##` 清洗、轻量 `@js:` 后处理和结果归一化
5. 组装 `RemoteChapterContent`

说明：

- 第 10 节中的“清洗”和“后处理”是执行阶段语义，不代表这些逻辑散落在业务仓库类中
- 实际实现统一收敛在 `RuleExecutionEngine` 内部，业务流只负责调用该执行器

## 11. 错误处理与诊断

### 11.1 目标

避免“失败都变成空结果”。

### 11.2 错误类型分级

建议内部统一标记以下类型：

- `REQUEST_FAILED`
- `RULE_EMPTY_RESULT`
- `UNSUPPORTED_RULE_KIND`
- `SCRIPT_POST_PROCESS_FAILED`
- `INVALID_SOURCE_DEFINITION`

### 11.3 行为约束

- 搜索：单源失败不影响其他源
- 详情 / 目录 / 正文：失败要能区分是站点问题还是兼容不足
- 对 `<js>`、登录态、复杂聚合源要明确识别为“当前阶段不支持”
- `SCRIPT_POST_PROCESS_FAILED` 统一视为兼容问题，不归类为站点请求失败

## 12. 首批 10 个目标源策略

### 12.1 筛选条件

从现有 `书源.json` 中筛出候选源，要求：

- `bookSourceUrl` 为标准 `http/https`
- 存在 `searchUrl`
- 存在 `ruleSearch`
- 存在 `ruleToc`
- 存在 `ruleContent`
- 不依赖 `<js>` 搜索模板
- 不需要登录态
- 主规则命中第一批支持范围

### 12.2 名单构成建议

为了避免只做一类站点，首批 10 个目标源建议分布为：

- 至少 4 个 HTML/CSS 主链源
- 至少 2 个 XPath 主链源
- 至少 2 个 JSON 搜索或 JSON 目录源
- 至少 2 个带 `##` 或轻量 `@js:` 的源

### 12.3 验收标准

每个目标源至少满足：

- 能搜到目标书
- 能打开详情
- 能拉到目录
- 能拉到一章正文

边角字段如封面、分类、最新章不要求第一批全部完美，但主链必须通。

## 13. 测试策略

### 13.1 解析层单测

验证规则拆解器是否正确识别：

- `$`
- `//`
- `##`
- `@js:`

### 13.2 执行层单测

用 fixture 验证：

- CSS 抽取
- XPath 抽取
- JSONPath 抽取
- 文本清洗
- 轻量脚本后处理

### 13.3 集成测试

对首批目标源建立离线 fixture，覆盖：

- 搜索
- 详情
- 目录
- 正文

### 13.4 App 级烟雾测试

至少补 1 到 2 条 app 级 smoke，验证兼容后的真实源可以从发现页进入阅读器并展示正文。

## 14. 实施顺序

建议按以下顺序推进：

1. 定义结构化规则模型
2. 实现规则拆解器
3. 实现统一执行器
4. 先打通搜索链路的 HTML 和 JSON 两条分支
5. 再补详情 / 目录 / 正文
6. 接入 `##` 和轻量 `@js:`
7. 选定首批 10 个目标源并补 fixture
8. 补 app 级烟雾测试
9. 写阶段文档和验收清单

## 15. 风险与缓解

### 风险 1：第一批目标源选得太复杂

缓解：

- 先按规则特征筛候选
- 把 `<js>`、登录态、聚合脚本源提前排除

### 风险 2：`@js:` 范围不断膨胀

缓解：

- 第一批只支持字段级后处理
- 不开放网络和运行时对象

### 风险 3：兼容逻辑再次散回仓库类

缓解：

- 强制把规则拆解和执行统一收敛到独立组件

### 风险 4：测试变成纯语法测试，缺少真实阅读闭环

缓解：

- 用目标源清单作为阶段验收基线
- 强制要求主链回归测试

## 16. 本阶段成功标准

第一批完成时，应同时满足以下条件：

- 结构化规则模型已落地
- `HTML/CSS`、`XPath`、`JSONPath` 三条取值链统一进入正式执行流
- `##` 和轻量 `@js:` 已进入正式执行流
- 首批 10 个目标源中，大多数可以跑通 `搜索 -> 详情 -> 目录 -> 正文`
- 第一批不支持的规则类型能被明确诊断
- 已补单测、fixture 集成测试和至少 1 到 2 条 app 烟雾测试

## 17. 后续阶段预留

为避免后续上下文压缩丢失，这里明确记录后续阶段要继续推进的方向：

1. `<js>` 搜索模板支持
2. 登录态 / cookie / 用户书架类源
3. Explore 发现页脚本源
4. 多页目录抓取
5. 多页正文拼接
6. 更复杂的 `@js:` 运行时能力
7. 聚合脚本服务类源兼容
8. 书源健康度、能力标签和自动筛选工具
