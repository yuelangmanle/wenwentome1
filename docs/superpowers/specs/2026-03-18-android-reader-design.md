## 一、目标与约束

### 1.1 产品目标

构建一个面向个人自用场景的安卓阅读器，统一支持本地书籍阅读与网文阅读，让二者在同一套书库、详情页、阅读器和同步体系中运作。

第一阶段的核心不是把所有能力一次做满，而是先把下面三件事做稳：

1. 本地书导入后能立即进入统一书库并顺畅阅读
2. 网文书目可以通过发现页加入统一书库，而不是形成独立系统
3. GitHub 私有仓库可以同步书文件与元数据，支持换机恢复

### 1.2 第一阶段优先级

- 第一优先级：书库管理体验
- 第二优先级：统一阅读体验
- 第三优先级：网文能力可用并可扩展

### 1.3 明确约束

- 平台：Android
- 技术路线：原生 Kotlin + Jetpack Compose
- 开发环境：GitHub Codespaces + GitHub Actions
- 使用场景：个人自用、侧载为主
- 本地格式第一阶段：TXT + EPUB
- 本地格式第二阶段：PDF
- 同步后端第一阶段：GitHub 私有仓库
- 同步范围第一阶段：书文件 + 书架元数据 + 进度/书签/设置
- 网文能力方向：内置书源 + 规则导入
- 网文兼容方向：尽量兼容阅读系现有生态，但以桥接层隔离复杂性

### 1.4 非目标

第一阶段不追求以下内容：

- 公开分发合规方案
- 完整 PDF 体验
- 复杂聚合发现页的全量兼容
- 重登录流程和高度依赖登录态的全部书源能力
- 所有规则格式一次性完全兼容
- iOS 或多端统一工程

- `Legado / 开源阅读`：参考现代阅读系规则能力边界与书源生态

- `搜书大师`：参考旧式成熟阅读产品的交互组织、旧规则模型与功能组合方式

### 2.2 借鉴内容

- 规则引擎边界：搜索、详情、目录、正文、Cookie、规则执行
- 书源管理思路：启用/禁用、分组、导入、导出、规则调试
- 阅读器基础能力：目录、书签、进度、阅读设置、缓存入口
- 旧模型经验：来源模型与阅读模型分离，而不是把网文数据直接塞进 UI 层

## 三、核心设计结论

### 3.1 采用“书库中台型”结构

本产品不采用“双系统拼合型”方案。

也就是说，本地书和网文不会分别形成一套书架、详情页、阅读器和同步逻辑，而是统一进入同一套书库模型，再进入同一套阅读与同步体系。

这个结论的原因是：

1. 用户明确将“书库管理”列为第一优先级
2. 用户希望这是一个真正的“二合一”产品，而不是两个系统并排摆放
3. 如果底层继续分裂，后续在筛选、搜索、同步、进度恢复和换机体验上会越来越割裂

### 3.2 BookRecord 是第一公民

顶层对象不是书源，也不是远端链接，而是“书库中的一本书”。

这意味着：

- 一本书必须先成为统一书目对象，才能进入书库
- 书源只是书的来源能力，不是顶层主对象
- 同步也以书和阅读状态为中心，而不是以来源为中心

### 3.3 网文能力通过桥接层接入

为了既保住自有产品架构，又不从零重写阅读系规则能力，网文部分采用桥接层方式接入。

即：

- 主工程：只处理统一书库、阅读、同步和产品交互
- `Source Bridge`：负责书源、规则、Cookie、远端抓取、规则兼容
- 旧阅读系能力：只作为桥接层设计参考或可复用能力来源

## 四、信息架构

### 4.1 顶层导航

底部保留三个主入口：

1. `书库`
2. `发现`
3. `我的`

### 4.2 书库

书库是默认首页，是产品第一主界面。

职责：

- 展示统一书架
- 提供导入入口
- 提供筛选、排序、搜索、分组、最近阅读、批量管理
- 显示来源徽标，但不按来源拆成两套书架

原则：

- 用户打开 app 先看到“我的书”，而不是推荐页
- 来源信息降级为标签，而不是结构级入口

### 4.3 发现

发现页只负责“找书”和“入库”。

职责：

- 网文搜索
- 内置源浏览
- 规则导入
- 书源管理
- 发现页内容浏览

限制：

- 发现页不是长期消费首页
- 发现页的目标动作是“加入书库”

### 4.4 我的

“我的”承担低频但关键能力：

- GitHub 私有仓库同步
- 备份与恢复
- 阅读设置
- 存储与缓存管理
- 规则调试、日志导出、实验功能

### 4.5 非底栏页面

以下页面不进入底栏，而是作为深层页面存在：

- 书籍详情页
- 阅读页
- 目录页
- 书源编辑/规则调试页

## 五、模块边界

### 5.1 自己实现的模块

#### App Shell

负责：

- 底部导航
- 页面跳转
- 权限与导入入口
- 全局主题
- 页面恢复与错误提示

原因：

这是产品感最强的一层，不能受旧项目结构牵制。

#### Library Core

负责：

- 统一书目模型
- 书架组织
- 分组/筛选/排序/搜索
- 最近阅读/收藏/归档/批量管理

原因：

第一阶段目标明确是“书库管理优先”，所以该层必须围绕你的产品目标设计。

#### Reader UX

负责：

- 统一详情页
- 统一阅读页
- 目录、书签、进度
- 阅读主题与排版设置

原因：

这是用户第二感知最强的层，而且后续会持续演进。

#### Sync Engine

负责：

- GitHub 私有仓库认证
- 同步清单管理
- 文件上传下载
- 冲突处理与恢复
- 设备迁移

原因：

这是产品的关键约束之一，现成旧项目并不会天然满足。

### 5.2 桥接层

#### Source Bridge

负责：

- 书源导入
- 规则解析与执行
- 搜索 / 详情 / 目录 / 正文获取
- Cookie / 登录态管理
- 来源兼容与异常隔离

原则：

- 向上只暴露统一接口，不把旧规则模型扩散到整个主工程
- 向下吸收复杂性，让书库层只看到标准化对象

### 5.3 仅借鉴或局部复用的能力

- 阅读系旧规则字段语义
- 旧书源兼容经验
- JSoup / XPath / JSONPath / JS 规则执行边界

不直接继承：

- 老 UI
- 广告/社区/支付系统
- 旧安卓大应用结构

## 六、数据模型骨架

### 6.1 BookRecord

统一书库中的一本书，是产品中的核心主对象。

建议字段：

- `id`
- `title`
- `author`
- `originType`：`local / web / mixed`
- `primaryFormat`：`txt / epub / web`
- `cover`
- `summary`
- `bookshelfState`
- `createdAt`
- `updatedAt`

意义：

- UI 和同步都围绕它运作
- 来源差异不直接影响书库主模型

### 6.2 BookAsset

表示与书关联的实际文件或缓存对象。

建议字段：

- `bookId`
- `assetRole`：正文文件 / 封面 / 缓存包
- `storageUri`
- `mime`
- `size`
- `hash`
- `syncPath`

意义：

- 支持 GitHub 仓库中文件同步
- 支持恢复和校验

### 6.3 ReadingState

统一表示阅读进度和阅读行为。

建议字段：

- `bookId`
- `locator`
- `chapterRef`
- `progressPercent`
- `bookmarks`
- `notes`
- `updatedAt`

意义：

- 让本地书和网文都能共享一套进度与书签机制
- 为冲突合并提供时间基准

### 6.4 SourceDefinition

书源定义对象，由桥接层管理，不应成为顶层产品对象。

建议字段：

- `sourceId`
- `sourceName`
- `sourceType`：内置 / 导入 / 兼容源
- `ruleFormat`：`legacy / legado3 / custom`
- `authState`
- `enabled`
- `group`

### 6.5 RemoteBinding

表示某本书与某个网文来源之间的绑定关系。

建议字段：

- `bookId`
- `sourceId`
- `remoteBookId`
- `remoteBookUrl`
- `tocRef`
- `syncMode`
- `lastFetchedAt`

意义：

- 把“书本身”与“来源绑定”区分开
- 同一本书理论上可以挂多个远端来源

### 6.6 SyncSnapshot

同步快照，用于仓库重建、冲突比较和设备恢复。

建议字段：

- `snapshotId`
- `revision`
- `manifest`
- `deviceId`
- `mergedAt`

## 七、关键用户流程

### 7.1 本地导入入库

目标：

让用户完成一次导入后，立即获得“能管理、能阅读、能同步”的结果。

流程：

1. 从书库页发起导入或扫描目录
2. 解析 TXT / EPUB 元数据和章节结构
3. 生成 `BookRecord + BookAsset + ReadingState`
4. 加入统一书库
5. 后台进入同步队列

### 7.2 网文加入书库

目标：

让网文不是“停留在发现页的临时对象”，而是成为书库中的正式书目。

流程：

1. 用户在发现页搜索或浏览来源
2. 桥接层获取详情与目录
3. 用户点击加入书库
4. 生成 `BookRecord + RemoteBinding`
5. 回到书库，以统一书目形式展示

### 7.3 GitHub 恢复与换机

目标：

让换设备时不需要重新手工导入全部书目。

流程：

1. 配置 GitHub 私有仓库
2. 拉取同步清单
3. 重建书库索引
4. 按需恢复文件、封面、书签、进度和设置
5. 用户直接从上次位置继续读

## 八、错误处理原则

### 8.1 导入错误

必须明确区分：

- 文件损坏
- 格式不支持
- 编码识别失败
- 元数据提取失败

### 8.2 书源错误

必须明确区分：

- 源失效
- 登录失效
- 规则不兼容
- 网络失败
- 正文解析失败

### 8.3 同步错误

必须明确区分：

- 仓库认证失败
- 仓库不可达
- 文件冲突
- 文件过大或上传失败
- 恢复不完整

原则：

- 不只告诉用户“失败了”
- 告诉用户失败在哪一层，以及可执行的下一步

## 十一、当前设计结论

目前已经确认的关键结论如下：

1. 产品采用“书库中台型”结构，而不是双系统拼合
2. `BookRecord` 是第一公民
3. 网文能力通过 `Source Bridge` 进入主工程
4. 第一期以本地书主链路为主，但网文必须可接入统一书库
5. GitHub 私有仓库同步是第一阶段正式能力，而不是以后再补
6. 第一阶段要重点做好书库管理，而不是优先做推荐或复杂发现页
