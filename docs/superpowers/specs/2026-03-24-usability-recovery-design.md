## 一、背景

`v1.5` 已经完成云端打包、签名发版与一轮本地书修复，但真实使用仍暴露出一批“可以发版但不好用”的问题：

1. `TXT` 导入后首次打开耗时明显，进入阅读页后出现“暂无正文内容”或等效空白状态
2. `EPUB` 导入存在失败案例，用户无法判断是文件真的坏了，还是当前解析器过严
3. 阅读页仍然把正文与目录 / 设置 / AI / 模式面板长期堆在同一屏，沉浸感很差
4. “我的”页首屏长期暴露 GitHub 同步表单；`API 中心` 首页也偏后台化，普通用户难以理解
5. 网文发现页缺少明确搜索提交动作，键盘回车不能直接搜索；书源管理只有导入和启用开关，没有删除能力

这些问题中，有些属于功能故障，有些属于信息架构和交互层级不对。用户本轮明确要求优先走“能用性救火”，而不是再做一次完整重构。

## 二、已确认决策

### 2.1 本轮优先级

- 选择：`先把能用性救回来`
- 先修功能闭环，再收口最影响使用的页面结构
- 不做大范围导航重构，不推翻现有 `Compose + Room + DataStore + SourceBridge` 骨架

### 2.2 这轮必须解决的五个用户痛点

1. 本地 `TXT` 导入后首开慢且正文空白
2. `EPUB` 导入失败
3. `API 中心` 首页难懂，“我的”页同步配置暴露过多
4. 阅读页主体太小，功能面板不该默认常驻
5. 网文发现页没有明确搜索提交动作，书源不能删除

### 2.3 交付策略

- 路线：`可用性抢修 + 轻量结构收口`
- 第一批优先抢修：
  - 本地书导入和阅读链路
  - 阅读页正文占比
  - 发现页搜索和书源管理
- 第二批再收口：
  - “我的”页同步入口层级
  - `API 中心` 首页认知负担

## 三、方案对比

### 方案 A：先做“功能抢修 + 关键页面轻改”（推荐）

内容：

- 保留现有数据模型和导航主干
- 修 `TXT / EPUB` 真实故障
- 把阅读页、发现页、“我的”页和 `API 中心` 做一轮层级调整

优点：

- 能最快把“导入后能读、搜索后能搜、页面能顺手用”落地
- 风险可控，不会在用户当前已经可安装使用的版本上引入大面积回归
- 适合立即进入测试和新包验证

缺点：

- 视觉层不会一步到位
- `API 中心` 仍保留高级配置体系，只是会被下沉，不是完全重做

### 方案 B：完整重做信息架构和主要页面

优点：

- 用户体验理论上最统一
- 可以一次性重做“我的 / API / 阅读 / 发现”四条主线

缺点：

- 改动面过大
- 会拖慢对 `TXT / EPUB` 故障的止血
- 需要重新验证更多导航和状态恢复场景

### 方案 C：只修 bug，不动页面结构

优点：

- 实施速度最快

缺点：

- 解决不了“正文太小”“同步表单常驻”“API 首页看不懂”这些核心体验问题
- 用户需要继续忍受错误的信息层级

结论：采用 `方案 A`。

## 四、本轮目标与非目标

### 4.1 本轮目标

本轮必须同时达成以下结果：

1. 本地 `TXT` 导入成功后能在合理时间内进入正文，且不再落到空白内容页
2. 尽可能兼容“结构脏但可读”的 `EPUB` 文件，导入失败时给出更清楚的错误语义
3. 阅读页默认以正文为核心，目录 / 设置 / AI / 模式全部改为按需唤起
4. 发现页搜索框支持明确按钮提交和键盘回车提交
5. 书源管理支持删除，至少能把不需要的源从本地配置中移除
6. “我的”页不再默认常驻同步大表单，`API 中心` 首页改成普通用户能理解的总览页

### 4.2 非目标

本轮不追求以下内容：

- 完整重做全站视觉语言
- 重写阅读器翻页引擎
- 重写本地书数据库模型
- 重写 `API Hub` 的 Provider / Binding / Budget / Price 全套高级页
- 重做整个底栏导航结构
- 扩展书源兼容范围或新增复杂书源语法

## 五、故障修复设计

### 5.1 TXT：首开慢与空正文

现状判断：

- `TXT` 导入层只是把原始字节落盘，不在导入阶段解析正文
- 真正的编码识别、切章、正文提取发生在首次阅读时
- 这意味着“导入成功”并不代表“首次进入阅读时就一定能正常解析”
- 当前阅读页遇到 `paragraphs.isEmpty()` 时直接显示“暂无正文内容”，把“解析失败”和“正文真的为空”混成了同一种表现

本轮设计：

1. 保留“原文件落盘 + 运行时解析”的主结构，不新增预解析数据库
2. 优化 `TXT` 首次解析：
   - 补强 UTF-8 / GB18030 识别后的兜底逻辑
   - 对章节切分失败但整本书有文本的场景，必须稳定回退到“正文全文段落”
   - 对空白行、段落切分和异常编码文本做更稳妥的归一化
3. 优化阅读态错误表达：
   - “正文为空”与“正文解析失败”必须区分
   - 对解析异常，阅读页显示明确失败文案，而不是误导性的“暂无正文内容”
4. 优化首开体感：
   - 优先保证首屏尽快出现正文
   - 避免不必要的重复解析和无意义等待

成功标准：

- 至少在 `UTF-8 TXT`、`GBK/GB18030 TXT`、无章节纯正文 `TXT` 三类样本上稳定可读
- 首开不再出现“卡很久进去以后是空内容”

### 5.2 EPUB：导入失败

现状判断：

- 当前 `EpubBookParser` 会在导入时执行一轮“可读性校验”
- 这类校验可以拦真正坏文件，但也可能误杀“结构不规范、却仍然可读”的 EPUB

本轮设计：

1. 保留“完全非法文件要拒绝导入”的原则
2. 放宽当前过严校验：
   - 区分“压根不是 EPUB / 无法解压 / 无核心资源”与“结构脏但仍能读”
   - 只要后续 `EpubCatalogParser / LocalBookContentRepository` 有能力读到正文，就不在导入阶段提前拦截
3. 导入失败文案清晰化：
   - 如果是真坏文件，明确告诉用户这是“文件无效或已损坏”
   - 不再把所有失败都收敛成同一种技术化异常

成功标准：

- 现有测试样本和更多真实 `EPUB` 文件能够导入
- 保留对明显坏文件的拒绝

## 六、阅读页收口设计

### 6.1 当前问题

`ReaderScreen` 当前把正文、目录、设置、AI、模式按钮和展开后的面板都放在主屏结构里，导致：

- 正文可视区域被明显压缩
- 阅读入口不像阅读器，更像调试面板
- 用户每次进入阅读都要先忍受功能区占位

### 6.2 新结构

本轮阅读页改成“正文优先”的层级：

1. 默认主视图：
   - 章节标题
   - 进度摘要
   - 正文区域
2. 次级功能：
   - 模式切换
   - 目录
   - 设置
   - AI
   改为点击后再展开的底部面板、浮层或抽屉

### 6.3 实施边界

- 不重做 `ReaderUiState` 和 locator 体系
- 不重写仿真翻页、横向翻页、纵向滚动三套正文呈现逻辑
- 只重排布局层级与面板唤起方式，确保正文默认占绝对主体

成功标准：

- 默认进入阅读页时，屏幕主要空间属于正文
- 所有功能面板均为按需调出，不默认常驻

## 七、发现页与书源管理设计

### 7.1 搜索提交方式

当前 `DiscoverScreen` 的搜索框只有 `onValueChange = onSearch`，意味着输入行为与搜索行为耦合在一起，且没有明确的提交动作。

本轮设计：

1. 搜索输入与搜索提交分离
2. 增加明确“搜索”按钮
3. 键盘回车触发搜索提交，而不是作为普通换行
4. 保留输入中状态，但不再把每次输入变动都等同为用户已经发起正式搜索

成功标准：

- 用户点击按钮可搜索
- 用户按回车可搜索
- 搜索行为可预测

### 7.2 书源删除

当前 `SourceManagementScreen` 只有：

- 导入
- 启用/禁用开关

缺少删除能力，导致错误导入、重复导入或无用书源无法清掉。

本轮设计：

1. 书源管理增加删除入口
2. 删除语义为“从本地源列表中移除该源”
3. 启用/禁用保留，删除是新增能力，不替代开关
4. 删除入口默认放在每条书源项的显式操作区，避免误触

成功标准：

- 用户可删除不需要的书源
- 删除后列表立即刷新

## 八、“我的”页与 API 中心收口设计

### 8.1 “我的”页

用户截图已经明确指出当前问题：同步配置表单整块长期暴露在首页，而且字段文案过于工程化。

本轮设计：

1. “我的”页首页只保留高频入口
2. GitHub 同步改成“云同步”入口卡片
3. 点击后再进入配置页或展开详情，不再默认常驻五个字段输入框
4. 字段文案改成人话：
   - `Owner` -> `GitHub 用户名`
   - `Repo` -> `仓库名`
   - `Branch` -> `分支`
   - `Bootstrap Token` -> `GitHub Token`
   - `Sync Password` -> `同步密码`

成功标准：

- “我的”页首屏明显更干净
- 新用户能看懂这些字段是干什么的

### 8.2 API 中心

当前 `ApiHubOverviewScreen` 更像一个后台总览页，而不是普通用户的功能入口。

本轮设计：

1. 首页改成“状态总览 + 新手入口”
2. 首屏只强调三件事：
   - 有没有配置可用 Provider
   - 当前能力绑定是否可用
   - 最近有没有错误 / 今日有没有调用
3. 高级项继续保留，但下沉到二级页：
   - Provider 管理
   - 模型绑定
   - 预算与回退
   - 价格目录
   - 调用记录

成功标准：

- 用户第一次进入 `API 中心` 时能知道“这是配置 AI 的地方”
- 高级配置仍可访问，但不在首屏造成认知轰炸

## 十、测试与验收

### 10.1 自动化验证

本轮至少补以下验证：

1. `TXT`：
   - UTF-8 文件能读到正文
   - GBK/GB18030 文件能读到正文
   - 无章节文本能回退到完整正文
2. `EPUB`：
   - 结构脏但可读的文件可导入
   - 明显坏文件仍会失败
3. 阅读页：
   - 默认不展开目录 / 设置 / AI / 模式面板
4. 发现页：
   - 明确搜索提交动作存在
   - 回车会触发搜索
5. 书源管理：
   - 删除后列表状态正确刷新

### 10.2 人工验收

用户侧手测至少覆盖：

1. 导入一个中文 `TXT`，确认能快速进入正文
2. 导入一个真实 `EPUB`，确认能入库和阅读
3. 进入阅读页，确认正文为主，功能通过点击再展开
4. 发现页输入书名，按回车直接搜索
5. 导入书源后删除一个无用源
6. 查看“我的”页和 `API 中心` 首页，确认首屏认知负担下降

## 十一、风险与回退

### 11.1 风险

- `TXT` 真实异常样本可能比当前测试覆盖更复杂
- `EPUB` 放宽校验后，可能引入“能导入但部分书仍然不好读”的中间态
- 阅读页结构收口如果改动过猛，可能影响现有测试和 locator 保存行为

### 11.2 回退原则

- 优先保住“导入可读”
- 若界面层收口与功能修复发生冲突，先保证功能闭环，再做视觉/层级细化
- 不在同一轮顺手扩额外需求，避免边做边膨胀
